# Prompt 15.5.3 — Fix BU pre-filter matching only one BU per change

> **Character.** Diagnose and fix. Server logs show only one `buatlas_invoke_start` per run for scenarios that should exercise multiple BUs. This is the parallelism showcase — if only one BU ever participates, the demo's most important architectural moment never happens.
>
> **How to use.** Paste below the `---` line into Claude Code.
>
> **Expected duration:** 20-30 minutes depending on which of three causes it turns out to be.

---

# Instructions for Claude Code

Server logs from a fresh run of scenario 006 (`change_006_multi_bu_affected_vs_adjacent.json`) show exactly **one** `buatlas_invoke_start` event, with `bu_id=bu_zeta`. This fixture is explicitly designed to affect multiple BUs — the filename says so. Scenario 001 also evaluates only one BU (`bu_alpha`). The pipeline is working, but the BU pre-filter is returning only one candidate BU instead of multiple.

Your job: diagnose which of three root causes is in play, fix it, and verify scenario 006 now evaluates at least 2 BUs (ideally 3 including ADJACENT).

## Environment discipline

`.venv/bin/python`, `.venv/bin/pytest`, `.venv/bin/ruff`, `.venv/bin/mypy`.

## Diagnosis — three hypotheses

### Hypothesis 1: SignalScribe's impact_areas output is too narrow

SignalScribe may be producing a ChangeBrief with only 1-2 impact_areas, and those only happen to overlap with one BU. The fix is in the SignalScribe prompt or the fixture.

### Hypothesis 2: bu_registry.yaml's owned_product_areas are disjoint

Each BU owns unique areas with no overlap. Whichever BU owns the specific area named in impact_areas wins, and the others are excluded. Fix is in `config/bu_registry.yaml`.

### Hypothesis 3: The pre-filter logic is buggy

It might be returning the first match and stopping, or using the wrong intersection logic. Fix is in the pre-filter code.

## Step 1 — Gather evidence

Run scenario 006 fresh (or look at the most recent audit log for it):

```
ls -t audit/$(date +%Y-%m-%d)/ | head -5
```

Get the change_id for the 006 run (UUID starts with `f6a7b8c9...` in the user's logs).

Read the audit log for that change:

```
cat audit/$(date +%Y-%m-%d)/<change_id>.jsonl | head -50
```

**Find these specific records:**

1. **SignalScribe output record** — what did `impact_areas` contain? Copy the full list.
2. **BU pre-filter output** — if there's an audit record for it, what BUs did it return?  
   If there's no explicit record, search the orchestrator code for the pre-filter call site (`grep -rn "pre_filter\|bu_candidates\|owned_product_areas" src/pulsecraft/orchestrator`) and add a debug log or temporarily print the candidate list for diagnosis.
3. **bu_registry.yaml** — open it. List each BU's `owned_product_areas`.

## Step 2 — Correlate

Compare:

- SignalScribe's impact_areas list (from step 1)
- Each BU's owned_product_areas (from bu_registry.yaml)

Which BUs *should* match? (Intersection is non-empty.)

If SignalScribe says impact_areas is `["analytics_portal", "reporting_dashboard"]` and:
- bu_alpha owns `["prior_authorization", "formulary_lookup"]` → no match (correct)
- bu_beta owns `["analytics_portal", "self_service_tools"]` → **should match** but didn't
- bu_zeta owns `["reporting_dashboard", "exports"]` → matched (correct)

Then the pre-filter is buggy. If all other BUs correctly don't overlap, the fixture and registry are the problem.

## Step 3 — Fix based on which hypothesis

### If Hypothesis 1 (SignalScribe too narrow):

Open `.claude/agents/signalscribe.md`. Find the part of the prompt that describes impact_areas. It likely says "name the product areas affected." The prompt is probably correct — the issue is that for some changes, only one or two product areas are genuinely affected.

**The right fix isn't to widen SignalScribe artificially.** If the change truly only affects one area, that's correct behavior.

In this case, the fix is in the **fixture** — `change_006_multi_bu_affected_vs_adjacent.json` needs to describe a change whose impact_areas naturally span multiple BUs' owned areas. Check the fixture; it may already list multiple areas in its change_summary but SignalScribe's extraction is too conservative.

If the fixture is broad but SignalScribe extracts narrowly: tune the fixture's language to be more explicitly multi-area. Example: explicitly mention "affects the reporting_dashboard and analytics_portal and self_service_tools."

### If Hypothesis 2 (disjoint owned_product_areas):

Open `config/bu_registry.yaml`. Review each BU's `owned_product_areas`. The design intent from CLAUDE.md is that BUs have overlapping areas — different BUs own adjacent slices of the same product surface.

Example of good overlap:
```yaml
bu_alpha:
  owned_product_areas:
    - prior_authorization
    - specialty_pharmacy_workflow
    - reporting_dashboard  # shared with zeta

bu_beta:
  owned_product_areas:
    - analytics_portal  # shared with zeta
    - hcp_portal_ordering
    - prescriber_attestation

bu_zeta:
  owned_product_areas:
    - analytics_portal  # shared with beta
    - reporting_dashboard  # shared with alpha
    - exports
```

Fix the YAML so that **the impact_areas from fixture 006 match 2-3 BUs**. Specifically: if fixture 006's SignalScribe output is `["analytics_portal", "reporting_dashboard", "exports"]`, then at minimum bu_beta and bu_zeta should both own `analytics_portal` or `reporting_dashboard` so both are candidates.

### If Hypothesis 3 (pre-filter logic bug):

Find the pre-filter: `grep -rn "def.*pre_filter\|def.*bu_candidates" src/pulsecraft/`.

Read it. Common bugs:
- Returns after first match (`return [bu]` inside a loop)
- Uses `any()` in a short-circuit context
- Case-sensitivity mismatch (e.g., `analytics_portal` vs `Analytics_Portal`)
- Substring matching accidentally wrong

Fix the logic. Should be: return *all* BUs whose `owned_product_areas` have a non-empty intersection with `impact_areas`.

## Rules for this session

- **Diagnose before fixing.** Don't guess. Read the audit log, the registry, the pre-filter code.
- **Fix the root cause, not a symptom.** If the registry is the problem, fix the registry. Don't force SignalScribe to output more areas.

- **Verify with live server run.** Don't report done without confirming ≥2 buatlas_invoke_start events in the server log.
