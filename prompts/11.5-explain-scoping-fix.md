## Design principles

1. **Single run is the sensible default.** 95% of the time operators asking "explain this change" want to know what happened *this time*, not the history. Design for the common case.
2. **Full history stays available.** Don't remove it — just make it opt-in.
3. **Run boundaries are inferred from audit structure.** A new run starts with a `state_transition` audit record where `output_summary` matches `"None → RECEIVED"`. Run ends with the next state transition into a terminal state (DELIVERED, ARCHIVED, HELD, AWAITING_HITL, SCHEDULED, FAILED).
4. **Schema addition is non-breaking.** `usd_estimate` added as optional field. Older audit records without it still validate.
5. **No behavior change for `--all`.** Current full-chain output still works — just not the default.
6. **Don't over-engineer run boundary detection.** Simple heuristic (next RECEIVED starts new run) is good enough for v1. Real edge cases — interleaved concurrent runs, partial failures mid-run — are future concerns.

### Step 2 — Run boundary detection

Add a function to `src/pulsecraft/skills/explain_chain.py`:

```python
from dataclasses import dataclass
from datetime import datetime
from typing import Literal

@dataclass
class RunBoundary:
    run_index: int  # 1-based from earliest
    start_timestamp: datetime
    end_timestamp: datetime
    terminal_state: str
    record_indices: tuple[int, int]  # start inclusive, end inclusive, into the full audit list

def detect_runs(records: list[AuditRecord]) -> list[RunBoundary]:
    """Split an append-only audit chain into logical runs.

    A run starts at a state_transition with output_summary matching 'None -> RECEIVED'
    and ends at the state_transition into the next terminal state (DELIVERED, ARCHIVED,
    HELD, AWAITING_HITL, SCHEDULED, FAILED) — or at the next RECEIVED if no terminal
    was reached.
    """
```

Terminal states to detect:
- `DELIVERED`
- `ARCHIVED`
- `HELD`
- `AWAITING_HITL`
- `SCHEDULED` (when no subsequent transition exists — treats scheduled-but-not-delivered as terminal for the purposes of this view)
- `FAILED`
- Any state transition that's followed by a fresh `None -> RECEIVED` implicitly ends the prior run

Implementation: iterate audit records; whenever you see `output_summary` starting with `None` and transitioning to `RECEIVED`, that's a new run start. Whenever you see a transition TO one of the terminal states, close the open run.

Edge cases:
- Empty audit list → empty list of boundaries
- Records that aren't state transitions sit inside the open run; they don't open or close a run
- If audit chain starts without a `None -> RECEIVED` (shouldn't happen, but defensive), treat from index 0 as run 1 start

Write unit tests in `tests/unit/skills/test_detect_runs.py` — at least:
- Single complete run → one RunBoundary
- Two sequential runs → two boundaries, non-overlapping
- Run that errored mid-pipeline (no terminal) → boundary closes at the next RECEIVED or at end of records
- Today's actual case — use the real audit file `audit/2026-04-22/a1b2c3d4-e5f6-4a7b-8c9d-0e1f2a3b4c5d.jsonl` → expect ~35 runs detected (use this as acceptance criteria; the exact number is whatever's actually in the file after today)
